# CLAUDE.md

## Rules

- **Python env**: Always use `.venv/bin/python` (not system `python3`).
- **Commits**: No `Co-Authored-By` lines. Single-line messages (no body).
- **Dependencies**: Managed in `requirements/*.txt` (used by local dev and Docker).
- **Docs sync**: When modifying code, update CLAUDE.md and ARCHITECTURE.md to reflect changes.
- **No memory**: Never use the auto-memory system (no MEMORY.md, no memory files). All persistent context belongs in CLAUDE.md or ARCHITECTURE.md.
- **Error handling**: App should crash on unexpected errors. `try/except` only for expected, recoverable errors. Custom exceptions in `exceptions.py`.
- **No API backward compat**: Project has no external users yet â€” don't preserve old Python APIs, function signatures, or import paths. Rename, delete, and rewrite freely; no shims or re-export modules. DB schema changes still go through Django migrations as normal â€” existing installs must upgrade cleanly.

## Project Overview

OpenOutreach â€” self-hosted LinkedIn automation for B2B lead generation. Playwright + stealth for browser automation, LinkedIn Voyager API for profile data, Django + Django Admin for CRM (models owned by this project).

## Architecture (quick reference)

For detailed module docs, see `ARCHITECTURE.md`.

- **`linkedin_cli`** (external package â€” [`eracle/linkedin-cli`](https://github.com/eracle/linkedin-cli), installed from PyPI as `linkedin-agent-cli` via `requirements/base.txt`, unpinned so builds get the latest auto-published release; import name `linkedin_cli`): Django-free library of LinkedIn platform mechanics (browser nav/login, Voyager API, profile/conversation scrape, connect/message/status/thread verbs), imports with no Django. Exposes a verb CLI â€” the installed `linkedin-cli` console command (or `python -m linkedin_cli.cli`): `session open/close`, `login`, `whoami`, `search`, `profile`, `status`, `connect`, `message`, `thread` â€” over a **bind+connect** transport â€” a session owner `browser.bind()`s the browser, clients `chromium.connect()`. **Output contract** (in the `cli.py` docstring so it travels with the package): each verb returns a result dict, rendered as a brief human summary by default or the full dict with `--json` (redirect with `>` to save; no `--out`); errors â†’ stderr + non-zero exit. The daemon owns its browser in-process and consumes the library directly; launch/cookie-persistence glue stays in `linkedin/browser/launch.py`. **Auth is a page-state machine** (`page_state.py`: `classify_page` judges the page by URL *path* only, `@transition(when=,then=)` contracts enforce pre/post state raising `IllegalPageTransition`, `PageFlow` is the generic observeâ†’act engine; `auth.py`: the login flow as declared `@auth_flow.transition` actions + `authenticate()`, shared by the CLI verb and the daemon â€” no duplicated login path). CRM/campaign/scheduling/ML stay in this project. (Extracted to its own repo + submodule; OpenOutreach consumes it as a dependency.)
- **Entry**: `manage.py` â€” stock Django management. `rundaemon` command (migrate â†’ onboard â†’ validate â†’ task queue loop). `manage.py` with no args defaults to `rundaemon`. Onboarding logic in `onboarding.py`: `OnboardConfig` (pure dataclass), `missing_keys()`, `collect_from_wizard()`, single `apply()` write path. The interactive wizard is self-contained â€” `onboarding_wizard.py` (questionary/prompt_toolkit step engine with back/skip/cancel) and `onboarding_prompts.py` (`SELF_HOSTED_QUESTIONS`), both vendored from the retired openoutreach-cli; no external `openoutreach` dependency. Docker `start` script handles Xvfb/VNC, then `exec python manage.py rundaemon`.
- **State machine**: `linkedin_cli/enums.py:ProfileState` â€” QUALIFIED â†’ READY_TO_CONNECT â†’ PENDING â†’ CONNECTED â†’ COMPLETED / FAILED. Deal.state is a CharField with ProfileState choices (no Stage model). `Outcome` (converted/not_interested/wrong_fit/no_budget/has_solution/bad_timing/unresponsive/unknown) on Deal.outcome. `Lead.disqualified=True` = permanent exclusion. LLM rejections = FAILED Deals with wrong_fit outcome (campaign-scoped).
- **Task queue**: `Task` model (persistent). Three types: `connect`, `check_pending`, `follow_up`. Handlers in `linkedin/tasks/`, signature: `handle_*(task, session, qualifiers)`. Task rows are **lazy**: `payload = {"campaign_id": <id>}` only; the handler resolves the concrete target at execution time via a single eligibility query. Slot creation is centralized in `linkedin/tasks/scheduler.py` â€” no other module inserts Task rows. The planner exposes one function per type (`plan_connect_window`, `plan_follow_up_window`, `plan_check_pending_window`); each emits `1 immediate + (n-1) Poisson-spaced` slots over the next 24h working window. `working_seconds_in_window(start, end)` and `poisson_slot_times(now, n, horizon_hours)` are the spacing primitives (uniform order-statistic implementation, exactly `n` slots, mean spacing `T/(n+1)` in working time). State-transition hook `on_deal_state_entered(deal)` only stamps `Deal.next_check_pending_at` on PENDING (no Task row); all other transitions are no-ops. `reconcile(session)` runs on daemon idle: it recovers stale RUNNING tasks, then iterates campaigns Ã— planners and re-plans any type whose pending queue is empty. Active hours are a single contiguous daily window (`ENABLE_ACTIVE_HOURS`, `ACTIVE_START_HOUR`, `ACTIVE_END_HOUR`, `ACTIVE_TIMEZONE`) â€” no weekend skip. `CHECK_PENDING_DAILY_CAP` (default 100) bounds the per-cycle check_pending slot count. On 401 (`AuthenticationError`), the daemon calls `session.reauthenticate()` and marks the task FAILED; the planner picks the type up on the next idle cycle.
- **ML pipeline**: GPR (sklearn) + BALD active learning + LLM qualification. Per-campaign models stored in `Campaign.model_blob` (DB).
- **Config**: `SiteConfig` DB singleton (LLM_PROVIDER, LLM_API_KEY, AI_MODEL, LLM_API_BASE â€” editable via Django Admin; `llm_provider` chooses between OpenAI/Anthropic/Google/Groq/Mistral/Cohere/openai_compatible, `llm_api_base` only consulted when provider is `openai_compatible`), `conf.py:CAMPAIGN_CONFIG` (timing/ML defaults), `linkedin_cli/conf.py` browser constants (`BROWSER_*`, `HUMAN_TYPE_*`, `BROWSER_HEADLESS`), `conf.py` schedule constants (`ENABLE_ACTIVE_HOURS` flag, active hours/timezone/rest days), `conf.py` onboarding defaults (`DEFAULT_*_LIMIT`), `conf.py:FASTEMBED_CACHE_DIR` (persistent model cache, defaults to `<project>/.cache/fastembed/`), Campaign/LinkedInProfile models (Django Admin). `VOYAGER_REQUEST_TIMEOUT_MS` lives in `linkedin_cli/api/client.py` (constructor default on `PlaywrightLinkedinAPI`). `linkedin_cli/conf.py:DUMP_PAGES` (default `False`) â€” enable to save page HTML snapshots for fixture collection.
- **Lazy accessors**: `Lead.get_profile(session)` is a pure live Voyager scrape (no DB caching of the raw dict); `Lead.get_urn(session)` reads the `urn` column and falls back to a scrape; `Lead.get_embedding(session)` lazily scrapes + embeds on first access, then caches the 384-dim bytes on the row. `Lead.embed_from_profile(profile)` reuses an in-hand profile dict to skip the scrape (used by `create_enriched_lead`). `Lead.to_profile_dict()` returns a minimal `{lead_id, public_identifier, url, meta}` dict (no `profile` key). `AccountSession.campaigns` (cached_property, list). `AccountSession.self_profile` (cached_property, re-discovers via Voyager on first access per session â€” no DB cache).
- **Deal summaries**: `Deal.profile_summary` and `Deal.chat_summary` are lazy, mem0-style JSON fact lists built on demand and updated incrementally. `linkedin/db/summaries.py` is the single boundary â€” `materialize_profile_summary_if_missing(deal, session)` fires on the first follow-up touch (one Voyager re-scrape per `(lead, campaign)` lifetime), `update_chat_summary(deal, new_messages, *, seller_name)` folds newly-synced ChatMessages into the summary via `reconcile_facts`, which routes new facts through mem0's UPDATE prompt to apply ADD/UPDATE/DELETE/NONE events (no naive append-and-dedup). **Only incoming (lead) messages reach fact extraction** â€” outgoing seller messages are filtered at the boundary so `chat_summary` stores facts about the lead, never the seller's pitch. A one-sided burst of outgoing messages short-circuits the LLM call entirely. **Identity binding (required):** `extract_facts`, `update_chat_summary`, and `reconcile_facts` all require a `seller_name` kwarg, sourced via `seller_name_from(session)` (first_name from `session.self_profile`, username fallback) and injected into the system prompt so the LLM stops inferring `the lead's name is Diego` from a lead reply like `"Hola Diego, gracias..."`. The mem0 reconcile prompt also receives the binding and is instructed to DELETE stored facts that describe the seller as if they were the lead â€” so previously-contaminated rows *should* be cleaned up on the next sync that produces a conflicting fact, though this is best-effort (mem0's vendored prompt is example-heavy and the cleanup hint is one prepended sentence). Dormant deals with no new messages stay contaminated until eager cleanup. The follow-up agent consumes `profile_summary + chat_summary + last 6 ChatMessage rows` instead of flat profile fields; its prompt (`follow_up_agent.j2`) carries the same `Me`â†’`self_name` binding under the verbatim-messages header. The fact-extraction prompt is vendored at `linkedin/db/summaries.py:_FACT_EXTRACTION_PROMPT`; mem0's `DEFAULT_UPDATE_MEMORY_PROMPT` and `get_update_memory_messages` are vendored under `linkedin/vendor/mem0/configs/prompts.py` (mirroring upstream paths so future syncs are a clean diff). No `mem0ai` runtime dependency â€” avoids qdrant/grpcio/sqlalchemy transitive bloat.
- **Django apps**: `linkedin` (main â€” Campaign with users M2M), `crm` (Lead with embedding/Deal), `chat` (ChatMessage).
- **Unified inbox** (`MessageThreadAdmin`, M5): server-rendered admin inbox over `MessageThread`/`Message` â€” filter by account/unread/has-reply/lead-source, search lead+body, newest-first. **Privacy default** (`ToolScopeFilter`) shows only threads this tool messaged into unless `?scope=all`. Actions: mark-read (+ per-thread endpoint) and pause-campaign (active `LeadCampaignState`â†’`paused_manual`).
- **InMail** (`linkedin/actions/inmail.py`, M4): `send_inmail(session, lead, subject, body)` â†’ result dict. App-side Playwright composer (no `linkedin_cli` InMail verb). Gated by `LinkedInProfile.has_inmail` (Sales Nav/Recruiter) â€” without it the step **skips cleanly** (never crashes the sequence); UI errors are captured in the result, not raised. The M2 executor's InMail step calls this and logs only on success.
- **Reply detection** (`linkedin/inbox/poller.py`, M3): `MessageThread` (one per lead+account) + `Message` (direction in/out, idempotent by `linkedin_message_id`). `poll_replies(session)` fetches each active sequence lead's conversation, persists messages, and flips `LeadCampaignState` â†’ `stopped_reply` when an inbound message post-dates the lead's `last_action_at` â€” the non-negotiable "if they replied, stop messaging." `fetch_thread_messages` is the lone mockable boundary; direction/id are heuristic because `linkedin_cli.get_conversation` returns only `{sender,text,timestamp}`. Driven by `manage.py run_reply_poller` + daemon hook.
- **Sequence engine** (`linkedin/sequences/executor.py`, M2): HeyReach-style branching playbooks. `Sequence` owns a tree of `SequenceStep`s (`parent`+`branch` where `success`=accepted/replied, `failure`=not-accepted/no-reply; `step_type` âˆˆ connect/message/inmail/wait/profile_visit/like_post; `config` JSON per type). `Campaign` gains `sequence`/`lead_list`/`status` â€” a campaign with a `sequence` is **sequence-driven** (executor), else stays on AI-discovery. `LeadCampaignState` is the per-(lead,campaign) cursor (`current_step`, `awaiting_decision`, `next_action_due_at`, terminal `state`s, never deleted). `enroll_campaign` seeds states from the lead_list; `run_due_states` advances due/active states; connect is two-phase (send â†’ wait `wait_days_before_branch_decision` â†’ accepted?â†’success/failure). Browser verbs are mockable module wrappers (`send_connection_request`/`is_connection_accepted`/`send_message`/`send_inmail`/`visit_profile`/`like_recent_post`). Driven by `manage.py run_sequences` + a daemon-loop hook. InMail action is stubbed until M4.
- **Manual lead import** (`linkedin/leads/importer.py`): the manual counterpart to AI auto-discovery. `LeadList` (`linkedin.models`) is a named lead collection, soft-delete only (`archived_at`, never hard-deleted); `Lead.lead_list` is a nullable `SET_NULL` FK. `import_csv` dedups by `public_identifier`; `import_search_url` scrapes a people-search URL (paginating via the `page` param, 1000 cap, idempotent) and enriches via the existing Voyager path. Entry points: `manage.py leadlist_import --source csv|search_url` and the Django admin (LeadList list/detail + CSV-upload view, Lead filterable by enrichment status). Both leave the AI-discovery path untouched.
